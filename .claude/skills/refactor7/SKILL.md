# 重构 (Refactoring)

本skill提供35个工程级重构案例，每个案例都包含问题代码和重构后的代码对比。

## 核心哲学

### 清晰优先于一切
代码质量的首要标准是：**一个理性的工程师能否在短时间内准确理解这段代码在做什么，以及为什么这样做。**

- 可读性是代码的第一属性
- 性能、扩展性、抽象能力都不能以牺牲清晰度为代价
- 写代码的目标是减少后来者（包括未来的自己）的理解成本

### 复杂性是一种成本
- 每一层抽象、每一个技巧，都必须证明其必要性
- 无法解释清楚的复杂性即为不合格设计
- 不为未来假想需求提前设计复杂结构

### 代码是逻辑结构的直接映射
- 好代码的结构应当能直接反映问题的逻辑结构
- 读代码应当像读一棵清晰的逻辑树
- 优先通过结构和命名消除注释需求

## 重构原则

1. **降低认知负担** - 让代码更易读易懂
2. **消灭腐烂源头** - 消除重复和坏味道
3. **提升可维护性** - 使代码更容易修改和扩展
4. **增强可推理性** - 减少隐式依赖和副作用

## 强制要求

### 逻辑完整性
- 所有条件分支必须显式、完整
- 禁止"隐含假设"和依赖读者脑补
- 必须区分"正常但无结果"和"异常状态"

### 模块化
- 一个函数只做一件事
- 函数名必须能完整概括其行为
- 函数应能在一个屏幕内阅读完（通常 ≤ 40 行）

### 命名清晰
- 命名必须表达真实业务含义
- 禁止使用模糊或占位式命名（data、info、temp、flag）
- 禁止用同一个返回值表达多种语义

## 9大类别重构案例

根据用户的代码问题，选择对应的类别查看详细案例：

### A. 长函数 & 可读性（6个案例）
**目标：** 降低认知负担，是所有重构的入口

查看 [long-functions.md](references/long-functions.md) 了解：
- 长函数 → 提炼函数（顺序流程）
- 长函数 → 提炼函数（含 early return）
- 长函数 → 提炼函数（含异常处理）
- 嵌套层级过深 → 提炼条件函数
- 混合"计算 + IO" → 拆分纯函数
- 业务流程写成脚本 → 用命名函数表达意图

### B. 重复代码（4个案例）
**目标：** 消灭腐烂源头

查看 [duplicate-code.md](references/duplicate-code.md) 了解：
- if/else 中的重复逻辑 → 提炼公共函数
- 多处复制校验逻辑 → Validator 抽取
- 不同类中相似算法 → 模板方法
- 测试中的重复 setup → 测试夹具提炼

### C. 条件复杂 / 分支爆炸（5个案例）
**目标：** 减少 if-else 树

查看 [complex-conditions.md](references/complex-conditions.md) 了解：
- 类型判断 if-else → 多态
- 策略切换 if-else → Strategy
- 复杂布尔条件 → 语义化函数
- switch + 魔法数 → 枚举 + 行为
- 条件嵌套 → Guard Clause（提前返回）

### D. 参数与数据结构问题（4个案例）
**目标：** 接口稳定性

查看 [parameters.md](references/parameters.md) 了解：
- 参数列表过长 → 参数对象
- 多个 boolean 参数 → 枚举/配置对象
- Map<String, Object> → 明确类型对象
- 同时传入相关参数 → 引入 Value Object

### E. 临时变量 & 状态污染（3个案例）
**目标：** 提升可推理性

查看 [temp-variables.md](references/temp-variables.md) 了解：
- 临时变量滥用 → Replace Temp with Query
- 方法依赖外部可变状态 → 显式传参
- 函数修改入参 → 返回新对象（消副作用）

### F. 数据类 / 行为漂移（3个案例）
**目标：** 修复"伪 OO"

查看 [data-behavior.md](references/data-behavior.md) 了解：
- 工具类操作数据类 → 行为移回数据
- Getter/Setter 驱动逻辑 → 封装行为
- 贫血模型 → 领域方法内聚

### G. 异常与错误处理（3个案例）
**目标：** 明确失败路径

查看 [error-handling.md](references/error-handling.md) 了解：
- catch Exception 什么都不做 → 显式失败
- 返回 null 表示失败 → 异常或 Result
- 多层 try-catch → 异常边界下沉

### H. 命名 & 表达力（3个案例）
**目标：** 减少注释依赖

查看 [naming.md](references/naming.md) 了解：
- 魔法数 → 命名常量
- 注释解释逻辑 → 命名函数
- 模糊变量名（data / info）→ 领域名词

### I. 类职责过重 & 资源管理（4个案例）
**目标：** 为架构演进铺路 + 生产稳定性

查看 [class-responsibility.md](references/class-responsibility.md) 了解：
- God Class → Extract Class
- Service 同时做计算 + 持久化 → 分层
- Manager/Util 泛化类 → 明确子职责
- 手动资源关闭 → RAII / try-with-resources

## 使用方法

### 方式一：针对特定代码片段重构
1. **识别Code Smell** - 根据代码问题选择对应类别
2. **查看对比案例** - 阅读问题代码和重构后代码
3. **理解重构手法** - 学习重构的思路和原则
4. **应用到实际代码** - 根据案例对用户代码进行重构

### 方式二：全项目自动化重构（执行 /refactor7 或直接输入 "refactor7"）

**重要：当用户输入 "refactor7" 指令时，立即启动以下完整工作流程：**

#### 第一阶段：代码扫描与问题识别

1. **发现所有代码文件**
   ```bash
   # 使用 Glob 工具查找常见编程语言的源代码文件
   # 包括：*.py, *.js, *.ts, *.jsx, *.tsx, *.java, *.go, *.rb, *.php, *.cs, *.cpp, *.c, *.h 等
   # 排除：node_modules/, venv/, dist/, build/, .git/ 等目录
   ```

2. **逐文件分析代码质量**
   - 使用 Read 工具依次读取每个源代码文件
   - 根据35个重构案例的标准，分析每个文件是否存在以下问题：
     - A类：长函数（>40行）、深层嵌套（>3层）、混合计算与IO
     - B类：重复代码块、重复的验证逻辑
     - C类：复杂条件判断、多层if-else、switch魔法数
     - D类：参数过多（>3个）、多个boolean参数、Map传参
     - E类：临时变量滥用、外部状态依赖、修改入参
     - F类：贫血模型、getter/setter滥用、行为与数据分离
     - G类：空catch块、返回null表示错误、多层try-catch
     - H类：魔法数字、模糊命名（data/info/temp）、注释驱动
     - I类：God Class、职责混杂、资源未正确管理

3. **生成重构待办清单**
   - 使用 TodoWrite 工具创建详细的待办事项列表
   - 每个待办项必须包含：
     - 文件路径和行号（例如：`src/service.py:45-78`）
     - 问题类别（A-I）
     - Code Smell 类型（例如："长函数"、"重复代码"）
     - 优先级（高/中/低）：
       - 高：严重影响可读性和维护性的问题
       - 中：明显的代码异味但不阻塞理解
       - 低：可优化但非必需的改进
   - 按优先级排序待办事项

4. **向用户展示分析报告**
   ```
   📊 代码质量分析完成

   共扫描文件：X 个
   发现问题：Y 处

   问题分布：
   - A. 长函数 & 可读性：X 处
   - B. 重复代码：X 处
   - C. 条件复杂：X 处
   - D. 参数问题：X 处
   - E. 状态污染：X 处
   - F. 行为漂移：X 处
   - G. 错误处理：X 处
   - H. 命名问题：X 处
   - I. 职责过重：X 处

   详细清单已生成，准备开始重构。
   ```

5. **询问用户确认**
   - 使用 AskUserQuestion 询问：
     - 是否立即开始重构？
     - 是否只重构特定优先级的问题？
     - 是否只重构特定类别的问题？

#### 第二阶段：逐项重构实施

**仅在用户确认后执行此阶段。**

1. **按优先级顺序处理**
   - 从高优先级问题开始
   - 每次只处理一个待办项
   - 标记当前项为 `in_progress`

2. **执行重构步骤**
   对每个待办项：

   a. **读取相关代码**
      - 使用 Read 工具读取完整文件
      - 理解上下文和依赖关系

   b. **应用重构模式**
      - 根据问题类别，参考 references/ 下对应的案例
      - 选择最合适的重构手法
      - 使用 Edit 工具进行精确修改

   c. **验证修改**
      - 确保语法正确
      - 保持原有功能不变
      - 改进代码结构和可读性

   d. **记录修改**
      - 在待办项中添加修改说明
      - 标记当前项为 `completed`
      - 移至下一项

3. **处理测试（如果存在）**
   - 如果项目有测试文件，在重构后运行测试
   - 使用 Bash 工具执行测试命令（例如：`pytest`, `npm test`, `mvn test`）
   - 如果测试失败，回滚修改或修复问题

4. **生成重构报告**
   完成所有重构后，生成总结报告：
   ```
   ✅ 重构完成

   已完成：X/Y 项
   - 修改文件：Z 个
   - 应用手法：[列出使用的重构模式]

   主要改进：
   - 减少代码行数：X 行
   - 消除重复：X 处
   - 简化复杂度：X 处

   建议：
   - [如有未完成项，说明原因]
   - [如有需要人工决策的地方，列出]
   ```

#### 执行要求

- **必须使用 TodoWrite 工具** 管理整个重构流程
- **严格遵循核心哲学** - 清晰优先于一切
- **保持功能等价性** - 重构不改变外部行为
- **逐步小步前进** - 每次只做一个小改动，确保安全
- **优先处理高价值问题** - 先解决影响最大的代码异味
- **遇到不确定时使用 AskUserQuestion** - 涉及业务逻辑判断时必须询问用户

#### 特殊情况处理

1. **遇到无法自动重构的代码**
   - 在待办项中标记为 "需要人工决策"
   - 说明原因（例如：涉及业务逻辑理解、需要重构覆盖范围太大）
   - 提供重构建议供用户参考

2. **遇到测试失败**
   - 立即停止当前重构
   - 分析失败原因
   - 如果是重构引入的问题，回滚修改
   - 如果是原有测试问题，标记并询问用户

3. **项目无测试覆盖**
   - 在报告中警告用户
   - 建议重构后进行人工测试
   - 可选：询问用户是否需要先添加测试
