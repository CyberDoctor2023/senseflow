# SenseFlow 开发实录：Vibecoding 的真实体验

**作者**: Jack
**项目**: SenseFlow - macOS 剪贴板管理器
**时间**: 2026年1月-2月（4周）
**开发模式**: Vibecoding with Claude Code

---

## 什么是 Vibecoding？

传统编程：我写代码 → 编译 → 调试 → 查文档 → 再写代码

Vibecoding：我描述想法 → Claude 实现 → 我看效果 → 调整 → 完成

**核心区别**：我专注于"产品应该是什么样"，Claude 负责"怎么实现"

### 动画的黑边问题

**问题**：
> "窗口消失的时候，卡片周围有黑边，文字也闪烁"

**Claude 的调研过程**（我能看到）：
```
Claude: 让我先尝试优化渲染...
Claude: 使用 .drawingGroup() 试试
Claude: 嗯，卡片流畅了，但窗口问题还在
Claude: 让我查查 WWDC 视频...
Claude: 没找到相关内容
Claude: 搜索社区案例...
Claude: 找到了！这是跨层动画问题
```

**Claude 的解释**：
> "问题是这样的：
> - Material 效果在 GPU 层渲染
> - SwiftUI 的 opacity 在 CPU 层计算
> - 两者不同步导致黑边
>
> 解决方案：
> - 窗口动画用 AppKit (NSAnimationContext)
> - 内容动画用 SwiftUI
> - 不要混用"

**我**：
> "我不太懂这些技术细节，但效果确实好了！"

**这个过程的启发**：
- Claude 不是"一次性给答案"
- 而是"和我一起调研、尝试、解决"
- 就像一个有经验的同事
- 即使我不懂技术细节，也能得到正确的解决方案

## Week 4：架构重构的决策

### 为什么要重构？

**背景**：我想添加 AI 工具功能（Prompt Tools）

**我尝试添加**：
> "我想加个功能，选中文字后能调用 AI 处理"

**Claude**：
> "可以实现，但我注意到代码有点耦合...
> 建议先重构一下架构，采用 Clean Architecture"

**我的疑虑**：
> "听起来很复杂，真的有必要吗？现在不是能用吗？"

**Claude 的解释**：
> "现在确实能用，但如果要添加新功能：
> - 业务逻辑和框架耦合，难以测试
> - 想支持多个 AI 服务商（OpenAI、Claude、DeepSeek）会很乱
> - 代码会越来越难维护
>
> Clean Architecture 的好处：
> - 业务逻辑独立，易于测试
> - 新增 AI 服务商只需实现接口
> - 代码更清晰，易于维护"

**我**：
> "好吧，试试看。但我不懂这些架构概念..."

**Claude**：
> "没关系，我会一步步引导你"

---

### 重构的过程

**第一步**：定义接口

**Claude**：
> "首先，我们定义需要什么能力：
> - AIServiceProtocol：调用 AI 服务
> - ClipboardReader：读取剪贴板
> - ClipboardWriter：写入剪贴板"

**我**：
> "嗯，这个我能理解"

**第二步**：实现用例

**Claude**：
> "然后，我们定义业务流程：
> 1. 读取剪贴板内容
> 2. 调用 AI 服务处理
> 3. 返回结果
> 4. 写入剪贴板（可选）"

**我**：
> "这就是我想要的功能！"

**第三步**：连接组件

**Claude**：
> "最后，用依赖注入把这些组件连接起来"

**我**：
> "依赖注入是什么？"

**Claude**：
> "简单说，就是让组件不直接创建依赖，而是从外部传入。
> 这样可以轻松替换实现，比如测试时用 Mock"

**我**：
> "哦，明白了"

---

### 重构的成果

**完成后**：
- 代码更清晰
- 添加新 AI 服务商很容易
- 有了单元测试

**我**：
> "确实比之前好维护了！而且我发现添加新功能更快了"

**Claude**：
> "是的，前期投入一些时间重构，后期会节省更多时间"

**这个过程的启发**：
- 重构不是"推倒重来"
- 而是"渐进式改进"
- Claude 会引导我理解架构概念
- 即使我不是架构专家，也能做出好的设计

**差距**：90 倍！
