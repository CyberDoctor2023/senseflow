### SwiftUI Performance: drawingGroup() GPU 加速
- **链接**: https://developer.apple.com/videos/play/wwdc2024/10151/
- **要点**:
  - 将视图层级合成为单个离屏图像（GPU 加速）
  - 适用场景：复杂视图的 hover/scale 动画
  - 参数：opaque: false（保留透明度）、colorMode: .linear（线性颜色空间）
  - ⚠️ 限制：无法用于包含 TextField 等原生控件的视图（会报错 "Unable to render flattened version"）

### NSPanel Window Animation Architecture (macOS)
- **链接**: https://developer.apple.com/documentation/appkit/nsanimationcontext
- **要点**:
  - ✅ 窗口级动画：使用 NSAnimationContext + window.animator().alphaValue
  - ❌ 避免：在 SwiftUI 层对包含 Material 的整个窗口做 opacity 动画（会产生黑边/闪烁）
  - 正确架构：AppKit 负责窗口淡入淡出，SwiftUI 负责内部元素动画
  - 示例：NSAnimationContext.runAnimationGroup({ context in window.animator().alphaValue = 1.0 })
  - timingFunction: CAMediaTimingFunction(controlPoints:) 可自定义缓动曲线

- **链接**: https://developer.apple.com/documentation/swiftui/view/drawinggroup(opaque:colormode:)
- **WWDC**: Session 10151 (2024) - Create custom visual effects with SwiftUI
- **要点**:
  - 将多层视图合成为单个离屏图像（GPU 纹理）
  - 适用场景: Material + Shadow + 复杂内容的卡片视图
  - 性能提升: 减少 ~70% 渲染通道，显著改善滚动流畅度
  - Material 效果已由 SwiftUI 自动 GPU 加速，无需手动优化
  - 配合 LazyHStack 懒加载，实现最佳性能
  - 动画参数: extraBounce 0.05 提供自然回弹感

## Context7 查询记录

### 2026-02-04: 多显示器窗口定位最佳实践
- **来源**: WebSearch + Context7 综合调研
- **要点**:
  - NSScreen.main 返回菜单栏所在屏幕，不是用户当前工作的屏幕
  - NSScreen.screens 返回所有连接的显示器数组
  - 检测活跃屏幕方法：
    1. 获取 frontmost application: NSWorkspace.shared.frontmostApplication
    2. 使用 CGWindowListCopyWindowInfo 获取该应用的窗口信息
    3. 根据窗口坐标判断在哪个屏幕上
  - 窗口尺寸动态调整：每次显示前调用 setFrame 重新计算尺寸
  - 最佳实践：setFrame 在 orderFront 之前调用，避免用户看到调整过程
- **应用**: FloatingWindowManager 需要在每次 showWindow 时检测活跃屏幕并调整窗口尺寸

### 2026-02-04: Clean Architecture - 可选功能的依赖注入模式
- **来源**: pointfreeco/swift-composable-architecture + nalexn/clean-architecture-swiftui
- **要点**:
  - Use Case 依赖接口（Protocol），不依赖具体实现或 UserDefaults
  - 可选功能检查应在 Adapter 层处理，不在 Use Case 层
  - 推荐模式：创建 `AutoPaster` 协议，Adapter 内部检查 `auto_paste_enabled` 设置
  - Use Case 只调用 `autoPaster.performIfEnabled()`，不关心设置逻辑
  - 符合单一职责原则（SRP）和依赖倒置原则（DIP）
- **应用**: ExecutePromptTool 注入 AutoPaster 协议，实现自动粘贴功能

### 2026-02-04: UserDefaults 访问模式 - SwiftyUserDefaults vs @AppStorage
- **来源**: sunshinejr/swiftyuserdefaults + pointfreeco/swift-sharing
- **要点**:
  - @AppStorage 适合 SwiftUI View 层，自动触发视图更新
  - UserDefaults.standard 适合 Adapter 层，轻量级读取设置
  - 避免在 Use Case 中直接访问 UserDefaults（违反 Clean Architecture）
  - 可选：创建 SettingsProvider 协议封装设置访问（过度设计）
  - 推荐：Adapter 内部直接读取 UserDefaults（简单有效）
- **应用**: AutoPasteAdapter 内部读取 `auto_paste_enabled` 设置

### 2026-02-04: 多显示器窗口定位性能优化
- **来源**: Alfred 多屏配置 + NSWindow setFrame 性能分析
- **要点**:
  - NSWindow.setFrame() 会触发昂贵的窗口重绘和布局计算
  - 鼠标位置检测本身非常快（微秒级），不是性能瓶颈

  - 同一屏幕重复打开：使用 setFrameOrigin() 快速路径（无重绘）
  - 切换到不同屏幕：使用 setFrame() 完整路径（需要重绘）
- **应用**: FloatingWindowManager 缓存 lastScreen，避免不必要的窗口重绘

### 2026-02-04: SwiftUI 窗口显示性能优化综合方案
- **来源**: SwiftUI LazyHStack 性能 + 数据库后台线程模式 + macOS 预加载策略
- **要点**:
  - ScrollView .id() 重建是最大性能杀手：强制销毁并重建整个视图层级
  - 数据库查询必须在后台线程：使用 DispatchQueue.global(qos: .userInitiated)
  - 预加载策略：在动画开始前触发数据加载，并行执行
  - 通知时机：动画前发送通知（预加载），而非动画后（延迟显示）
  - SwiftUI .task vs .onAppear：.task 适合 async/await，GCD 场景用 .onAppear 即可
  - LazyHStack 性能：避免不稳定的 ID，保持视图轻量级
- **应用**: 删除 scrollViewID 重建、数据库查询后台化、实现预加载机制

### 2026-02-04: 多窗口同步淡入淡出动画实现
- **来源**: NSAnimationContext 文档 + NSHostingView 多实例模式 + macOS 窗口管理最佳实践
- **要点**:
  - NSAnimationContext.runAnimationGroup 可同步多个窗口动画
  - 使用 window.animator().alphaValue 实现淡入淡出
  - NSHostingView 每次创建都是独立实例，需共享 ViewModel
  - 实现模式：创建临时窗口 → 同步动画 → 销毁旧窗口
  - 关键代码：context.duration = 0.35, oldWindow.animator().alphaValue = 0.0, newWindow.animator().alphaValue = 1.0
  - 完成回调：动画结束后清理旧窗口，新窗口成为主窗口
- **应用**: FloatingWindowManager 跨屏幕切换时同步淡入淡出

### 2026-02-05: SwiftUI 窗口动画闪烁问题 - 社区案例分析
- **来源**: Web 搜索 - SwiftUI animation flicker, Alfred/Raycast 实现分析
- **要点**:
  - 图片闪烁：动画过程中转换图片格式会导致闪烁，应直接存储 UIImage

  - 使用 Instruments Core Animation 工具诊断渲染瓶颈
- **应用**: 缩短动画时长，改用 ease-out 曲线，使用 Instruments 诊断

### 2026-02-05: 本地图片同步加载性能问题
- **来源**: Apple SwiftUI 官方文档 - Image, AsyncImage
- **要点**:
  - `Image(nsImage:)` 是同步加载，会阻塞主线程
  - AsyncImage 仅适用于网络图片，不适用于本地文件
  - ⚠️ 问题：`item.getImage()` 从磁盘读取大图片（>512KB）时会阻塞主线程
  - 首次渲染时，多个卡片同时加载图片会导致明显卡顿
  - 建议：使用后台线程预加载图片，或实现图片缓存机制
  - 可选方案：使用 Task { } 异步加载图片，配合 @State 更新视图
- **应用**: 优化 ClipboardCardView 的图片加载逻辑，避免主线程阻塞

### 2026-02-05: SwiftUI 视图生命周期和数据加载时机
- **来源**: Apple SwiftUI 官方文档 - task, onAppear, View Lifecycle
- **要点**:
  - `.task` 在视图出现**之前**运行异步任务，但只提前几毫秒
  - `.onAppear` 在视图出现**之前**运行同步任务
  - `.task` 会在视图消失时自动取消任务，避免内存泄漏
  - ⚠️ 问题：`.task` 不会等待数据加载完成，视图会先显示空状态
  - 数据加载是异步的，窗口动画和数据加载并行执行
  - 如果数据加载时间 > 动画时间，用户会看到空状态→列表的突然切换
- **应用**: 需要在窗口显示前预加载数据，或优化数据加载速度

### 2026-02-05: SwiftUI 动画性能和渲染优化
- **来源**: Apple SwiftUI 官方文档 - Animation, LazyHStack, drawingGroup
- **要点**:
  - 动画选择：`.smooth` 适合窗口过渡，`.interactiveSpring` 适合交互响应
  - `.snappy` 适合快速状态变化，不适合窗口滑入等长时间动画
  - LazyHStack 首次渲染需计算可见视图几何信息，可能导致延迟
  - `.drawingGroup()` 将视图扁平化为离屏图像，首次渲染有 GPU 合成开销
  - ⚠️ 简单卡片视图使用 `.drawingGroup()` 可能反而降低性能
  - 建议：复杂视图（多层模糊、阴影）才使用 `.drawingGroup()`
- **应用**: 评估当前动画曲线和 `.drawingGroup()` 的使用是否合理

### 2026-02-10: GlassEffectContainer 分离/合并动画

### 2026-02-10: Liquid Glass 单窗口合并方案调研
- **来源**: Apple Developer Documentation - glassEffectUnion, GlassEffectContainer, Applying Liquid Glass
- **要点**:
  - **问题根因**: 两个 NSPanel 只能有一个是 key window，Liquid Glass 在非 key 窗口渲染质量降低
  - **`glassEffectUnion(id:namespace:)`**: 让多个视图贡献到同一个玻璃形状（相同 shape/variant/ID 合并）
  - **`GlassEffectContainer(spacing:)`**: spacing 控制形状融合距离，大于布局间距会导致静止时融合
  - **关键发现**: addChildWindow 共享 active 状态不够，Liquid Glass 区分 key vs non-key
  - **解决方案**: 合并为单个 NSPanel，搜索栏和卡片列表在同一窗口内用透明间隔分隔
  - **性能提示**: 限制 GlassEffectContainer 数量，避免在容器外过多使用 glassEffect
- **应用**: FloatingWindowManager 合并双窗口为单窗口架构

### 2026-02-11: Liquid Glass 焦点状态管理 - appearsActive 和 containerBackground
- **来源**: Apple SwiftUI 官方文档 - appearsActive, containerBackground, MaterialActiveAppearance
- **要点**:
  - **appearsActive 环境变量**: 控制视图是否显示活跃外观（macOS 10.15+）
    - 替代已弃用的 controlActiveState（macOS 26.2 弃用）
    - `.environment(\.appearsActive, true)` 强制活跃外观
  - **containerBackground vs background**:
    - `.containerBackground(.clear, for: .window)` 自动填充整个父容器
    - `.background()` 仅应用于视图本身，不影响窗口级背景
  - **Liquid Glass 背景干扰警告**: "Any custom backgrounds and appearances you use in these elements might overlay or interfere with Liquid Glass"
  - **NSWindow.AnimationBehavior.none**: 禁用系统自动动画，适合自定义窗口动画
  - **关键发现**: 灰色背景是因为窗口级背景未设置为透明，需用 containerBackground
- **应用**: UnifiedPanelView 使用 containerBackground + appearsActive 解决焦点问题

### 2026-02-11: NSWindow 动画状态管理最佳实践
- **来源**: Apple AppKit 文档 - NSWindow.isVisible, NSAnimationContext, orderOut
- **要点**:
  - **isVisible 行为**: 窗口在屏幕上时为 true（即使被遮挡），orderOut() 后才变 false
  - **动画期间状态**: NSAnimationContext 动画执行期间 isVisible 保持 true
  - **状态冲突场景**: 动画中失去焦点 → handleWindowResignKey 检查 isVisible=true → 重复触发 hideWindow
  - **解决方案**: 添加 isHiding 状态标志，防止动画期间重复触发隐藏
  - **NSAnimationContext 嵌套**: 支持多层嵌套，每层独立 duration 和 timingFunction
  - **completion handler 时机**: 动画完成后才调用，此时 orderOut() 已执行
- **应用**: WindowLifecycle 需添加 isHiding 标志，避免动画期间状态冲突
